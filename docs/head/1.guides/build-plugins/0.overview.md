**Quick Answer:** Each framework package ships a unified Vite plugin at `@unhead/{framework}/vite` that handles build optimizations. One import, one plugin call.

## Setup

Install your framework's unhead package (if you haven't already), then add the plugin to your Vite config:

::FrameworkCode

#vue
```ts [vite.config.ts]
import vue from '@vitejs/plugin-vue'
import { Unhead } from '@unhead/vue/vite'

export default defineConfig({
  plugins: [vue(), Unhead()],
})
```

#react
```ts [vite.config.ts]
import react from '@vitejs/plugin-react'
import { Unhead } from '@unhead/react/vite'

export default defineConfig({
  plugins: [react(), Unhead()],
})
```

#svelte
```ts [vite.config.ts]
import { svelte } from '@sveltejs/vite-plugin-svelte'
import { Unhead } from '@unhead/svelte/vite'

export default defineConfig({
  plugins: [Unhead(), svelte()],
})
```

#solid
```ts [vite.config.ts]
import solid from 'vite-plugin-solid'
import { Unhead } from '@unhead/solid-js/vite'

export default defineConfig({
  // Must come before solid() to see JSX before compilation
  plugins: [Unhead(), solid()],
})
```

#nuxt
```ts
// No configuration needed
```

::

## What Does It Do?

The plugin combines several build-time optimizations:

- **[Tree-shaking](/docs/head/guides/build-plugins/tree-shaking)**: Removes deprecated server composables and `useSchemaOrg` calls from client bundles
- **[useSeoMeta transform](/docs/head/guides/build-plugins/seo-meta-transform)**: Converts `useSeoMeta()` calls into raw `useHead()` calls at build time (~3kb savings)
- **[Minify transform](/docs/head/guides/build-plugins/minify-transform)**: Pre-minify static inline script/style content at build time
- **Static precompilation (experimental)**: Compile the sealed `unhead/precompiled/server` entry into render-ready plans
- **Dev validation**: Auto-injects `ValidatePlugin` in dev so head tag warnings surface in the browser console (enabled by default)
- **[Devtools](/docs/head/guides/build-plugins/devtools)**: Inspect head tags, entries, and SEO metadata in Vite DevTools (dev only, enabled by default)

## Options

```ts
Unhead({
  // Disable server composable tree-shaking
  treeshake: false,
  // Disable useSeoMeta → useHead transform
  transformSeoMeta: false,
  // Pre-minify inline script/style content
  minify: { js: createJSMinifier(), css: createCSSMinifier() },
  // Compile sealed static server entries at build time
  experimental: { precompile: true },
  // File filter (shared across all transforms)
  filter: { exclude: [/some-file/] },
})
```

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `treeshake` | `object \| false` | enabled | [Tree-shake server composables](/docs/head/guides/build-plugins/tree-shaking) from client bundles |
| `transformSeoMeta` | `object \| false` | enabled | [Transform `useSeoMeta()`](/docs/head/guides/build-plugins/seo-meta-transform) to `useHead()` |
| `minify` | `object \| false` | disabled | [Pre-minify static inline script/style](/docs/head/guides/build-plugins/minify-transform) |
| `experimental.precompile` | `boolean \| object` | disabled | Compile sealed `unhead/precompiled/server` calls into render-ready plans |
| `validate` | `boolean` | enabled | Auto-inject `ValidatePlugin` in dev for head tag warnings |
| `devtools` | `object \| false` | enabled | [Vite DevTools integration](/docs/head/guides/build-plugins/devtools) (dev only) |
| `filter` | `object` | — | Shared include/exclude file filter |
| `sourcemap` | `boolean` | — | Enable sourcemap generation for transforms |

The transformer options accept `false` to disable them entirely, or an object to configure them.

## Static Precompilation

Static precompilation is a sealed core SSR target. It moves normalization, CAPO weighting, identity, sanitization, attribute serialization, and tag serialization into the server build. Generated plans are hoisted at module scope, so a request only collects plan references, resolves execution-order duplicates, and joins strings.

```ts [vite.config.ts]
Unhead({
  experimental: {
    precompile: true,
  },
})
```

```ts
import { createHead, renderSSRHead, useHead } from 'unhead/precompiled/server'

const head = createHead()
useHead({
  title: 'Product',
  meta: [{ name: 'description', content: 'Static product page' }],
}, { head })

const payload = renderSSRHead(head)
```

The mode is compile-or-error. Unsupported syntax fails the build with its file, line, and reason rather than preserving a large fallback runtime.

| Supported | Rejected or excluded |
|---|---|
| JSON-compatible `useHead()` literals | Variables, functions, refs, getters, spreads, and computed keys |
| Static `useSeoMeta()` literals | Observed return values, `patch()`, and `dispose()` |
| Package-owned pre-normalized defaults | Raw `init`, plugins, hooks, custom `tagWeight`, and custom `propResolvers` |
| `disableDefaults`; compact rendering without line breaks | Other create/render options and entry options other than `{ head }` |
| Static tag priority and valid position | `titleTemplate`, invalid tag positions, `templateParams`, `processTemplateParams`, and custom duplicate strategies |
| Contiguous arrayable SEO metadata, grouped atomically per call | Explicit tag keys, class/style attributes, and interleaved structured media arrays |
| Plain HTML/body attributes | Framework server entries and streaming replay |

Use `unhead/server` or a framework server entry when you need any excluded capability.

The sealed renderer always omits formatting line breaks. This removes a per-fragment branch and produces the smallest response; line-break formatting is intentionally not dynamic in this mode.

Repeated arrayable identities must be adjacent inside one call. For example, a structured media array that emits `og:image`, then `og:image:width`, then another `og:image` is rejected because regrouping it would change render order. Split it into contiguous calls or keep that module on the normal server entry.

Only imports from `unhead/precompiled/server` are compiled. Imports from ordinary Unhead and framework entries stay unchanged, and client-targeted builds skip the phase. The sealed implementation therefore adds no code to ordinary or browser bundles.

Vite detects SSR builds automatically. For a plain Rollup build, or another bundler that cannot expose its target, opt into the server consumer explicitly with `experimental: { precompile: { consumer: 'server' } }`.

## Other Bundlers

Framework packages expose a unified bundler entry with per-bundler methods:

```ts
import { Unhead } from '@unhead/vue/bundler'

export default {
  plugins: Unhead().webpack(),
}
```
